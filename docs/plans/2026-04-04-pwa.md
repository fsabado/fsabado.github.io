# PWA Plan — fsabado.com portfolio

**Date**: 2026-04-04  
**Status**: Planned  
**Stack**: Astro 4 (static `output`), Tailwind, GitHub Pages (`dist/` → Pages artifact)  
**Production URL**: HTTPS at **`https://fsabado.com`** (apex canonical); **`www.fsabado.com` → 301 to apex** (see §2).

---

## 1. Goals

- Meet baseline **Progressive Web App** criteria so the site can be **installed** (where the browser supports it), **load faster on repeat visits** (cached shell/assets), and show **simple offline fallbacks** when the network is unavailable (not a full offline copy of every page).
- Keep the implementation **maintainable** inside this repo: manifest + icons + service worker integrated with the Astro build, not ad-hoc copies in `dist/`.
- Preserve **performance**: cache only what matters; avoid huge precache lists or stale UI after deploys.

**Non-goals (unless you explicitly want them later)**

- Native-app parity, push notifications, or background sync (possible with more work; out of scope for v1).

---

## 2. Hostname: `www` vs apex (decided)

`https://www.fsabado.com` and `https://fsabado.com` are **different origins**. A service worker registered on one **does not** apply to the other, and caches/install state won’t carry over.

**Decision**: **`https://fsabado.com` is canonical.** Configure **`www.fsabado.com` to 301-redirect** to `https://fsabado.com` (DNS / registrar / CDN / GitHub Pages custom domain — wherever you manage hostnames).

**Repo alignment**

1. **`astro.config.mjs`** — keep `site: 'https://fsabado.com'` (already matches).
2. **`BaseLayout.astro`** — canonical URLs already use `Astro.site` + pathname; no change needed for hostname choice.
3. **Deploy / DNS** — ensure the redirect from `www` → apex is live before relying on PWA install/cache behavior for real users.

Until redirects and `site` agree, treat edge cases (users landing on `www`) as **second-class** for SW scope; ideally everyone ends on apex after one redirect.

## 3. Deliverables checklist

| Item | Purpose |
|------|--------|
| **Web app manifest** | Name, short name, `start_url`, `scope`, `display`, theme/background colors, icon set |
| **Icons** | At least **192×192** and **512×512** PNGs derived from the **existing favicon** (`public/favicon.ico`); add **maskable** 512 optional |
| **`theme-color` / `meta`** | Matches manifest; already partially covered by manifest + layout |
| **Service worker** | Precache core assets; **simple offline page(s)** (e.g. dedicated `offline.html` or minimal fallback) for failed navigations; runtime cache as needed for repeat visits |
| **Registration** | One small client script that registers the SW (production-only is fine) |
| **Verification** | Chrome DevTools → Application; Lighthouse PWA; install smoke test on mobile |

## 4. Manifest

**File**: place in `public/` as `manifest.webmanifest` (or `site.webmanifest`) so it copies to site root at build time.

**Fields to set**

- `name` / `short_name` — portfolio title.
- `start_url`: `/` (with `base: '/'` this is correct).
- `scope`: `/` — must encompass all in-app URLs (blog, projects, etc.).
- `display`: `standalone` or `minimal-ui` (prefer `standalone` for “app-like” chrome).
- `background_color` / `theme_color` — align with your CSS theme (light/dark is tricky: pick a default or document that the shell uses one color until first paint).
- `icons`: array with `sizes`, `type`, `purpose` (`any` vs `maskable`).

**Wire-up**

- In `src/layouts/BaseLayout.astro`, add:

  ```html
  <link rel="manifest" href="/manifest.webmanifest" />
  <meta name="theme-color" content="#…" />
  ```

- Optional: `apple-touch-icon` links if you care about iOS home-screen appearance beyond Safari’s defaults.

---

## 5. Icons

**Source**

- Use the **existing icon**: `public/favicon.ico` (already linked in `BaseLayout.astro`). For PWA install prompts, **ICO alone is not enough** — export or rasterize to **192** and **512** PNGs (ImageMagick, Sharp, or a design tool).

**Process**

1. Generate **192**, **512**, and optionally **maskable 512** from `favicon.ico` (upscaling a tiny favicon may look soft; acceptable for v1 or regenerate from a higher-res source if you have one later).
2. Store under `public/icons/` (or similar) and reference in the manifest.

**Quality bar**

- Crisp edges at 512; test maskable safe zone so nothing important is clipped on Android launchers.

## 6. Service worker strategy

**Constraints**

- Static hosting: no server-driven cache headers beyond what GitHub Pages sends; the SW is the main lever for offline and installability signals.
- Astro emits **hashed filenames** for bundled assets in many cases — precache lists should come from the **build** (see implementation options), not hand-maintained paths.

**Recommended behaviors**

1. **Precache**: `index.html` shell or critical static routes you care about offline; CSS/JS chunks from the build manifest.
2. **Offline navigations**: on network failure, respond with a **small static offline page** (same branding, short message, “try again”) rather than a generic browser error — matches “install + faster repeat visits + simple offline pages.”
3. **Runtime cache** (optional): cache recently visited pages or static assets to speed repeat visits without precaching the entire site.
4. **Skip waiting / clients claim**: on deploy, prompt or auto-activate new SW so users aren’t stuck on old caches forever (`skipWaiting` + `clients.claim` with a version bump strategy).

**Implementation options (pick one)**

| Approach | Pros | Cons |
|----------|------|------|
| **`@vite-pwa/astro` (or Vite PWA plugin wired for Astro)** | Auto precache from build, workbox patterns, less manual SW code | Dependency + config learning curve; verify Astro 4 compatibility |
| **Hand-written `public/sw.js` + build step to inject file list** | Full control, minimal deps | You maintain cache versioning and precache lists |
| **Minimal SW** (network-first for HTML, cache-first for static assets) | Simple | Easier to get wrong; test offline paths carefully |

**Recommendation**: Start with **Vite PWA–style integration** if it builds cleanly on this repo; otherwise a **small Workbox-based** or **minimal manual SW** with a documented update policy.

**Registration**

- Add a **client-only** script (inline module or small `src/scripts/register-sw.ts` imported with `client:load`) that calls `navigator.serviceWorker.register('/sw.js')` in production builds only (optional: guard with `import.meta.env.PROD`).

## 7. Astro / build integration notes

- **`public/`** files are copied as-is; manifest and non-generated `sw.js` live here.
- If the SW is **generated** into `dist/` by a plugin, ensure the **deploy artifact** includes it (it should if emitted under `dist/` root).
- **`base: '/'`** — no path prefix; `start_url` and SW `scope` stay `/`.
- **Project subpages** (`/projects/...`, `/blog/...`) must fall under the same `scope` — they do with `/`.
- **Large legacy HTML** under `public/projects/` (copied in CI): decide whether to precache or runtime-cache; avoid precaching everything if those folders are big.

---

## 8. Security & headers (sanity)

- Manifest and SW must be served from **same origin** as the app (they will be on GitHub Pages).
- Avoid registering SW on `localhost` during dev unless the plugin handles it — use `astro preview` to test SW behavior.

---

## 9. Testing & sign-off

**Local / staging**

1. `npm run build && npm run preview` — verify manifest loads, SW registers, no console errors.
2. DevTools → **Application** → Manifest + Service Workers — check scope and “Update on reload” during iteration.

**Production**

1. Lighthouse (Chrome) **PWA** category — address failures (installability, HTTPS, manifest, SW).
2. Install from address bar / “Install app” where available; confirm **start URL** and **icons**.
3. **Airplane mode** test: open installed app, hit a few routes you expect to work offline.

**Regression**

- Theme toggle + `localStorage` — ensure SW caching doesn’t serve stale HTML that fights client theme.
- **Fuse.js** search — if fully client-side, offline may still work after first load; if it fetches JSON indexes, add those URLs to cache strategy.

---

## 10. Implementation order (suggested)

1. **Verify** apex is canonical and **`www` → apex 301** in production; confirm `site` in `astro.config.mjs` stays `https://fsabado.com`.  
2. Add **icons** + **manifest** + head links; deploy — validate manifest in DevTools (no SW yet).  
3. Add **service worker** + registration; tune precache/runtime.  
4. Run **Lighthouse** + mobile install test; document cache-bust/versioning for future you.  
5. Optional: short note in `README.md` on “how the PWA is built” (one paragraph).
